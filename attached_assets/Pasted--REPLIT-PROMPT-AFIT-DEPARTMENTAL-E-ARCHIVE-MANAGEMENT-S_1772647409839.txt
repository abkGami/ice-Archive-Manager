# REPLIT PROMPT — AFIT DEPARTMENTAL E-ARCHIVE MANAGEMENT SYSTEM (FRONTEND)

---

## PROJECT OVERVIEW

Build a complete, multi-page React.js frontend for the **Air Force Institute of Technology (AFIT), Kaduna** Departmental E-Archive Management System. This is an internal document management portal for an academic department at AFIT. The system must have three distinct user role experiences: **Administrator**, **Lecturer (Staff)**, and **Student**. Use dummy data for all interactions. No backend or API calls are needed. All state must be managed in React using useState and useContext.

This is a production-quality, institutional web application. It must look serious, polished, and professional, fitting for a Nigerian Air Force university. Think of the UI standard of a premium university student portal redesigned with modern design principles applied thoughtfully.

---

## TECHNOLOGY STACK

- React.js (with functional components and hooks only)
- React Router v6 for all page navigation
- Tailwind CSS for all styling (no inline styles, no CSS modules, no styled-components)
- Lucide React for all icons
- No emoji anywhere in the UI
- No gradients anywhere in the UI
- No external UI component libraries (no shadcn, no MUI, no Chakra)

/dashboard                     — Redirects to role-specific dashboard
/admin/dashboard               — Administrator dashboard
/admin/documents               — All documents view (admin)

All routes except /login must be protected. If a user accesses a protected route while not logged in, redirect to /login. If a logged-in user tries to access a route outside their role, redirect them to their own dashboard.

## GLOBAL UI COMPONENTS

### Navigation Items Per Role

**Administrator sidebar nav:**
- MAIN SECTION: Dashboard (LayoutDashboard icon), Documents (FolderOpen icon), Upload Document (Upload icon)
- ADMINISTRATION SECTION: User Management (Users icon), Audit Log (ClipboardList icon)

**Lecturer sidebar nav:**
- MAIN SECTION: Dashboard (LayoutDashboard icon), Documents (FolderOpen icon), Upload Document (Upload icon)

**Student sidebar nav:**
- MAIN SECTION: Dashboard (LayoutDashboard icon), Browse Documents (FolderOpen icon)

### Toast Notification System
Build a simple toast system (no library). Toasts appear top-right. Types: success (green), error (red), warning (amber), info (sky blue). Each has an icon (CheckCircle, XCircle, AlertTriangle, Info from Lucide), a title, a message, and an X close button. Auto-dismiss after 4 seconds with a progress bar that depletes across the bottom of the toast.

### Empty States
Every table and list must have a designed empty state when there is no data (e.g., when search returns no results). Empty state: a centered icon (in light gray), a heading "No documents found", a subtext "Try adjusting your search or filters", and optionally an action button. Never show a blank white space.

### Loading States
For any action that would have a loading period (login button click, form submission, etc.), show a spinner inside the button and disable it during loading. Simulate a 1.2 second loading delay using setTimeout before resolving.

### Confirmation Dialog
Delete actions and deactivation actions must trigger a confirmation modal before executing. The modal: white card, centered, "Are you sure?" heading, description text, Cancel button (outlined) and Confirm Delete button (filled Danger Red). The modal has a dark overlay.

## INTERACTION DETAILS

- All page transitions: use a simple fade-in (opacity 0 to 1, 200ms) when a new page mounts
- All card hover states: translate-y -2px with a shadow increase, 150ms transition
- All button hover states: 150ms color transition
- Sidebar active state transition: 150ms background and border-left transition
- Document drawer: slides in from right, 300ms ease-out. Overlay fades in separately.
- All form inputs: smooth border-color transition on focus (150ms), never use outline — replace browser outline with a custom Sky Blue border-color
- Table rows: background color transition on hover (100ms)
- Status badge colors must never be just text — always a pill with background and text color
- All action icon buttons must have a title attribute for tooltip accessibility

---

## RESPONSIVE BEHAVIOR

Design primarily for desktop (1280px and above). At 1024px–1280px, reduce sidebar to 220px. Below 1024px, collapse the sidebar to icon-only mode (48px width) with tooltips on hover showing the label. Below 768px, stack the layout to a mobile hamburger menu. You do not need to design the mobile view in detail, just ensure it does not break.

---

## SPECIFIC THINGS TO AVOID

- No emoji anywhere — use Lucide icons exclusively for all visual indicators
- No gradients — use flat solid colors only
- No border-radius above rounded-md (8px) on any element — this is an institutional system, not a consumer app
- No shadows heavier than shadow-md
- No animations that feel playful or bouncy — transitions must feel precise and professional
- No lorem ipsum text — use realistic AFIT-appropriate placeholder text
- No placeholder image tags — if an avatar or image is needed, render initials or icons instead
- Do not use any color outside the defined color system
- The word "portal" should never appear in the in-app UI text — use "archive", "system", or "repository" instead
